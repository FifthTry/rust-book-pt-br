Ento, por que estamos recebendo esses warnings(avisos)? Afinal, estamos construindo uma biblioteca com funes que se destinam a ser usadas pelos nossos usurios, no necessariamente por ns dentro de nosso prprio projeto, por isso no deveria importar que essas funes connect no sejam utilizadas. O ponto de cri-las que elas sero usadas por outro projeto, no o nosso.
Para entender por que esse programa invoca esses warnings(avisos), vamos tentar usar a biblioteca connect de outro projeto, chamando-a externamente. Para fazer isso, vamos criar um crate binrio no mesmo diretrio que o nosso crate de biblioteca inserindo um arquivo src/main.rs que contm esse cdigo:
Arquivo: src/main.rs
extern crate communicator;

fn main() {
    communicator::client::connect();
}

extern crate communicator;

fn main() {
    communicator::client::connect();
}
Usamos o comando extern crate para trazer o crate de biblioteca communicator para o escopo. Nosso pacote agora contm duas crates. Cargo trata src/main.rs como um arquivo raiz de um crate binrio, que separado do crate de biblioteca existente cujo arquivo raiz src/lib.rs. Esse padro bastante comum para projetos executveis: a maioria das funcionalidades est em um crate de biblioteca e o crate binrio usa esse crate de biblioteca. Como resultado, outros programas tambm podem usar o crate de biblioteca, e uma boa separao de responsabilidades.
Do ponto de vista de um crate fora da biblioteca communicator todos os mdulos que criamos esto dentro de um mdulo que tem o mesmo nome como do crate, communicator. Chamamos o mdulo de nvel superior de um crate de mdulo raiz.
Observe tambm que, mesmo que estejamos usando um crate externo dentro de um submdulo do nosso projeto, o extern crate deve entrar em nosso mdulo raiz (ento em src/main.rs ou src/lib.rs). Ento, em nossos submdulos, podemos consultar itens de crates externos como se os itens fossem mdulos de nvel superior.
Agora, nosso crate binrio apenas chama a funo connect da nossa biblioteca do mdulo client. No entanto, invocar agora cargo build nos dar um erro aps os warnings:
error `client` is private

error `client` is private
 --> src/main.rs:4:5

Ah ha! Este erro nos diz que o mdulo client privado, que o cerne das advertncias. tambm a primeira vez em que nos encontramos com os conceitos de pblico e privado no contexto do Rust. O estado padro de todos os cdigos em Rust privado: ningum mais tem permisso para usar o cdigo. Se voc no usar uma funo privada dentro do seu programa, como ele o nico cdigo permitido a usar essa funo, Rust ir avis-lo de que a funo no foi utilizada.
Depois de especificar que uma funo como client::connect pblica, no s ser permitida a nossa chamada para essa funo a partir de nosso crate binrio, mas o warning(aviso) de que a funo no utilizada ir desaparecer. Marcar uma funo como pblica permite ao Rust saber que a funo ser usada por cdigo fora do nosso programa. Rust considera que agora possvel que a funo esteja sendo usada. Assim, quando uma funo marcada como pblica, Rust no exige que seja usada em nosso programa e deixar de avisar que a funo no utilizada.
Fazendo uma Funo Pblica

Para dizer ao Rust que torne pblica uma funo, adicionamos a palavra-chave pub ao incio da declarao. Nos focaremos em corrigir o warning que indica client::connect no foi utilizado por enquanto, assim como o erro module `client` is private (mdulo `client` privado) do nosso crate binrio. Modifique src/lib.rs para tornar o mdulo client pblico, assim:

Arquivo: src/lib.rs

pub mod client;

mod network;

pub mod client;

mod network;

Regras de Privacidade

No geral, estas so as regras para a visibilidade do item:

  1. Se um item for pblico, ele pode ser acessado atravs de qualquer um dos seus mdulos pais.
  2. Se um item privado, ele s pode ser acessado por seu mdulo pai imediato e qualquer um dos mdulos filhos do pai.

Exemplos de Privacidade

Vejamos mais alguns exemplos de privacidade para obter alguma prtica. Crie um novo projeto de biblioteca e digite o cdigo da Listagem 7-6 no arquivo src/lib.rs desse novo projeto:

Arquivo: src/lib.rs

mod outermost {
    pub fn middle_function() {}

    fn middle_secret_function() {}

    mod inside {
        pub fn inner_function() {}

        fn secret_function() {}
    }
}

fn try_me() {
    outermost::middle_function();
    outermost::middle_secret_function();
    outermost::inside::inner_function();
    outermost::inside::secret_function();
}

mod outermost {
    pub fn middle_function() {}

    fn middle_secret_function() {}

    mod inside {
        pub fn inner_function() {}

        fn secret_function() {}
    }
}

fn try_me() {
    outermost::middle_function();
    outermost::middle_secret_function();
    outermost::inside::inner_function();
    outermost::inside::secret_function();
}
Lista 7-6: Exemplos de funes pblicas e privadas, alguns dos quais esto incorretos
Antes de tentar compilar este cdigo, tente um palpite sobre quais linhas na funo try_me ter erros. Em seguida, tente compilar o cdigo para ver se voc estava certo e leia sobre a discusso dos erros!
Olhando para os Erros

A funo try_me est no mdulo raiz do nosso projeto. O mdulo chamado outermost privado, mas a segunda regra de privacidade afirma que a funo try_me pode acessar o mdulo outermost porque outermost est no mdulo atual (raiz), bem como try_me.

A chamada para outermost::middle_function funcionar porque middle_function pblica e try_me est acessando middle_function atravs do seu mdulo pai outermost. Determinamos no pargrafo anterior que este mdulo acessvel.

A chamada para outermost::middle_secret_function causar um erro de compilao. middle_secret_function privado, ento a segunda regra se aplica. O mdulo raiz no nem o mdulo atual de middle_secret_function (que seria o outermost), nem um mdulo filho do mdulo atual de middle_secret_function.

O mdulo denominado inside privado e no tem mdulos filhos, portanto, ele s pode ser acessado pelo seu mdulo atual outermost. Isso significa que a funo try_me no tem permisso de chamar outermost::inside::inner_function ou outermost::inside::secret_function.

Reparando os Erros

Aqui esto algumas sugestes para alterar o cdigo na tentativa de corrigir os erros. Antes de tentar cada uma, tente adivinhar se ela ir consertar os erros e, em seguida, compile o cdigo para ver se voc est certo ou no, usando as regras de privacidade para entender o porqu.

  • E se o mdulo inside fosse pblico?
  • E se outermost fosse pblico e inside fosse privado?
  • E se, no corpo de inner_function, voc chamasse o ::outermost::middle_secret_function()? (Os dois dois pontos no incio significam que queremos consultar os mdulos a partir do mdulo raiz.)

Sinta-se livre para projetar mais experimentos que lhe vierem mente!
