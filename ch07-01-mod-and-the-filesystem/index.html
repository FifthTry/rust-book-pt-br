mod e o Sistema de Arquivos

Vamos iniciar o nosso exemplo de mdulo fazendo um novo projeto com o Cargo, mas em vez de criar um crate binrio, faremos um crate de biblioteca: um projeto que as outras pessoas podem puxar para os seus projetos como uma dependncia. Por exemplo, o crate rand discutido no Captulo 2, um crate de biblioteca que usamos como uma dependncia no projeto do jogo de adivinhao.

Criaremos um esqueleto de uma biblioteca que fornece algumas funcionalidades gerais de rede; nos concentraremos na organizao dos mdulos e funes, mas no nos preocuparemos com o cdigo que est dentro das funes. Chamaremos nossa biblioteca de communicator. Por padro, o Cargo criar uma biblioteca, a menos que outro tipo de projeto seja especificado: se omitimos a opo --bin, que temos usado em todos os captulos anteriores a este, nosso projeto ser um biblioteca:

$ cargo new communicator
\$ cd communicator

$ cargo new communicator
\$ cd communicator
Observe que Cargo gerou src/lib.rs em vez de src/main.rs. Dentro de src/lib.rs encontraremos o seguinte:
Arquivo: src/lib.rs
#[cfg(test)]
mod tests {
    #[test]
    fn it_works() {
        assert_eq!(2 + 2, 4);
    }
}

#[cfg(test)]
mod tests {
    #[test]
    fn it_works() {
        assert_eq!(2 + 2, 4);
    }
}
Cargo cria um teste de exemplo para nos ajudar a comear nossa biblioteca, em vez de o binrio Hello, world! que recebemos quando usamos a opo --bin. Olharemos a sintaxe #[] e mod tests no Usando super para Acessar um Mdulo Pai mais adiante neste captulo, mas por agora, deixe este cdigo na parte inferior de src/lib.rs.
Como no temos um arquivo src/main.rs, no h nada para ser executado pelo Cargo com o comando cargo run. Portanto, usaremos o comando cargo build para compilar o cdigo da nossa biblioteca.
Examinaremos diferentes opes para organizar o cdigo da sua biblioteca que sero adequados em uma variedade de situaes, dependendo da inteno do cdigo.
Definies do Mdulo

Para a nossa biblioteca de rede communicator, primeiro definiremos um mdulo chamado network que contm a definio de uma funo chamada connect. Cada definio de mdulo em Rust comea com a palavra-chave mod. Adicione este cdigo ao incio do arquivo src/lib.rs, acima do cdigo de teste:

Arquivo: src/lib.rs

mod network {
    fn connect() {
    }
}

mod network {
    fn connect() {
    }
}
Aps a palavra-chave mod, colocamos o nome do mdulo, network e, em seguida, um bloco de cdigo entre chaves. Tudo dentro deste bloco est dentro do namespace network. Neste caso, temos uma nica funo, connect. Se ns quisermos chamar essa funo do cdigo fora do mdulo network, ns precisaremos especificar o mdulo e usar a sintaxe do namespace ::, assim: network::connect() em vez de apenas connect().
Tambm podemos ter mltiplos mdulos, lado a lado, no mesmo arquivo src/lib.rs. Por exemplo, para ter mais um mdulo client que possui uma funo chamada connect , podemos adicion-lo como mostrado na Listagem 7-1:
Arquivo: src/lib.rs
mod network {
    fn connect() {
    }
}

mod client {
    fn connect() {
    }
}

mod network {
    fn connect() {
    }
}

mod client {
    fn connect() {
    }
}
Listagem 7-1: O mdulo network e o mdulo client definidos lado a lado em src/lib.rs
Agora, temos uma funo network::connect e uma funo client::connect. Estas podem ter funcionalidades completamente diferentes, e os nomes das funes no esto em conflito entre si porque esto em mdulos diferentes.
Nesse caso, como estamos construindo uma biblioteca, o arquivo que serve como ponto de entrada para construir nossa biblioteca src/lib.rs. No entanto, em relao a criao de mdulos, no h nada de especial sobre src/lib.rs. Poderamos tambm criar mdulos em src/main.rs para um crate binrio da mesma forma que ns criamos mdulos em src/lib.rs para o crate de biblioteca. Na verdade, podemos colocar mdulos dentro de mdulos, o que pode ser til medida que seus mdulos crescem para manter juntas funcionalidades relacionadas e separar funcionalidades no relacionadas. A escolha de como voc organiza seu cdigo depende do que voc pensa sobre a relao entre as partes do seu cdigo. Por exemplo, o cdigo client e a funo connect podem ter mais sentido para os usurios de nossa biblioteca se eles estivessem dentro do namespace network, como na Listagem 7-2:
Arquivo: src/lib.rs
mod network {
    fn connect() {
    }

    mod client {
        fn connect() {
        }
    }
}

mod network {
    fn connect() {
    }

    mod client {
        fn connect() {
        }
    }
}
Listagem 7-2: Movendo o mdulo client para dentro do mdulo network
No seu arquivo src/lib.rs, substitua as definies mod network e mod client pelas da Listagem 7-2, que possuem o mdulo client como um mdulo interno da network. Agora temos as funes network::connect e network::client::connect: novamente, as duas funes denominadas connect no conflitam uma com a outra porque elas esto em diferentes namespaces.
Desta forma, os mdulos formam uma hierarquia. O contedo de src/lib.rs est no nvel superior mais alto, e os submdulos esto em nveis mais baixos. Aqui est a nossa organizao quando pensada de forma hierrquica na Listagem 7-1:
communicator
  network
  client

communicator
  network
  client
E aqui est a hierarquia correspondente ao exemplo na Listagem 7-2:
communicator
  network
      client

communicator
  network
      client
Conforme a hierarquia mostrada na Listagem 7-2, client um filho do mdulo network em vez de um irmo. Projetos mais complicados podem ter muitos mdulos, necessrio organiz-los logicamente para mant-los sob controle. O que logicamente significa em seu projeto fica a seu critrio, e depende do que voc e os usurios da sua biblioteca pensam sobre o domnio do seu projeto. Use as tcnicas mostradas aqui para criar mdulos lado a lado e mdulos aninhados em qualquer estrutura que voc queira.
Movendo Mdulos para Outros Arquivos

Os mdulos formam uma estrutura hierrquica, bem parecida com outra estrutura computacional que voc conhece: sistemas de arquivos! Podemos usar o sistema de mdulos do Rust juntamente com vrios arquivos para dividir projetos Rust de forma que nem tudo resida em src/lib.rs ou src/main.rs. Para este exemplo, vamos comear com o cdigo em Listagem 7-3:

Arquivo: src/lib.rs

    fn connect() {
    }
}

    fn connect() {
    }

        fn connect() {
        }
    }
}

    fn connect() {
    }
}

        }

communicator

Se esses mdulos tivessem muitas funes, e elas estivessem se alongando muito, seria difcil percorrer esse arquivo para encontrar o cdigo com que queremos trabalhar. Como as funes esto aninhadas dentro de um ou mais blocos mod, as linhas de cdigo dentro das funes comearo a se alongar tambm. Estes seriam bons motivos para separar os mdulos client, network, e server de src/lib.rs e coloc-los em seus prprios arquivos.
Primeiro, substitua o cdigo do mdulo client por apenas a declarao do mdulo client, para que seu src/lib.rs se parea com o cdigo mostrado na Listagem 7-4:

Ainda estamos declarando o mdulo client aqui, mas ao substituir o bloco por um ponto e vrgula, estamos dizendo ao Rust para que procure, em outro local, o cdigo definido no escopo do mdulo client. Em outras palavras, a linha mod client; significa:
mod client {

mod client {

Arquivo: src/client.rs

Rust s sabe olhar em src/lib.rs por padro. Se quisermos adicionar mais arquivos para o nosso projeto, precisamos dizer ao Rust em src/lib.rs para procurar em outros arquivos; por isso que mod client precisa ser definido em src/lib.rs e no pode ser definido em src/client.rs.

  2. Mova o arquivo src/network.rs para o novo diretrio network e renomeie para src/network/mod.rs.
  3. Mova o arquivo de submdulo src/server.rs para o diretrio network.

\$ mv src/server.rs src/network

\$ mv src/network.rs src/network/mod.rs
\$ mv src/server.rs src/network

O motivo que Rust no ser capaz de reconhecer que server deveria ser um submdulo de network se o arquivo server.rs estiver no diretrio src. Para esclarecer o comportamento de Rust aqui, consideremos um exemplo diferente com a seguinte hierarquia de mdulos, onde todas as definies esto em src/lib.rs:
communicator
  client
  network
      client

communicator
  client
  network
      client
Neste exemplo, temos novamente trs mdulos : client, network, e network::client. Seguindo os mesmos passos anteriores para extrair mdulos em arquivos, poderamos criar src/client.rs para o mdulo client. Para o mdulo network, poderamos criar src/network.rs. Mas no seramos capazes de extrair o mdulo network::client para um arquivo src/client.rs porque ele j existe para o mdulo client de nvel superior! Se pudssemos colocar o cdigo para ambos os mdulos client e network::client no arquivo src/client.rs, Rust no teria nenhuma maneira de saber se o cdigo era para client ou para network::client.
Portanto, para extrair um arquivo para o submdulo network::client do mdulo network, precisamos criar um diretrio para o mdulo network em vez de um arquivo src/network.rs. O cdigo que est no mdulo network entra no arquivo src/network/mod.rs, e o submdulo network::client pode ter seu prprio arquivo src/network/client.rs. Agora o o nvel superior src/client.rs inequivocamente o cdigo que pertence ao mdulo client.
Regras dos Mdulos e Seus Arquivos

Vamos resumir as regras dos mdulos em relao aos arquivos:

  • Se um mdulo chamado foo no possui submdulos, voc deve colocar as declaraes para foo em um arquivo chamado foo.rs.
  • Se um mdulo chamado foo possui submdulos, voc deve colocar as declaraes para foo em um arquivo chamado foo/mod.rs.

Essas regras se aplicam de forma recursiva, ento, se um mdulo chamado foo tiver um submdulo chamado bar e bar no possui submdulos, voc deve ter os seguintes arquivos no seu diretrio src:

 foo
    bar.rs (contm as declaraes em `foo::bar`)
    mod.rs (contm as declaraes em `foo`, incluindo `mod bar`)

 foo
    bar.rs (contm as declaraes em `foo::bar`)
    mod.rs (contm as declaraes em `foo`, incluindo `mod bar`)
Os mdulos devem ser declarados no arquivo do mdulo pai usando a palavra-chave mod.
