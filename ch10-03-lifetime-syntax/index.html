Validating References with Lifetimes

Quandos falamos sobre refercias no Captulo 4, ns deixamos de fora um detalhe importante: toda referncia em Rust tem um lifetime, que o escopo no qual aquela referncia vlida. A maior parte das vezes tempos de vida so implcitos e inferidos, assim como a maior parte do tempo tipos so inferidos. Similarmente quando temos que anotar tipos porque mltiplos tipos so possveis, h casos em que os tempos de vida das referncias poderiam estar relacionados de alguns modos diferentes, ento Rust precisa que ns anotemos as relaes usando parmetros genricos de tempo de vida para que ele tenha certeza que as refernciais reais usadas em tempo de execuo sero definitivamente vlidas.

Sim, um pouco incomum, e ser diferente de ferramentas que voc usou em outras linguagens de programao. Tempos de vida so, de alguns jeitos, a caracterstica mais distinta de Rust.

Tempos de vida so um tpico grande que no podero ser cobertos inteiramente nesse captulo, ento ns vamos cobrir algumas formas comuns que voc pode encontrar a sintaxe de tempo de vida nesse captulo para que voc se familiarize com os conceitos. O Captulo 19 conter informaes mais avanadas sobre tudo que tempos de vida podem fazer.

Tempos de Vida Previnem Referncias Soltas
O principal alvo de lifetimes prevenir referncias soltas, quais fazem com que o programa referencie dados quais ns no estamos querendo referenciar. Considere o programa na Listagem 10-18, com um escopo exterior e um interior. O escopo exterior declara uma varivel chamada r com nenhum valor inicial, e o escopo interior declara uma varivel chamada x com o valor inicial de 5. Dentro do escopo interior, ns tentamos estabelecer o valor de r como uma referncia para x. Ento, o escopo interior acaba, e ns tentamos imprimir o valor de r:
{
    let r;

    {
        let x = 5;
        r = &x;
    }

    println!("r: {}", r);
}

{
    let r;

    {
        let x = 5;
        r = &x;
    }

    println!("r: {}", r);
}

Variveis No Inicializadas No Podem Ser Usadas

Os prximos exemplos declaram vaiveis sem darem a elas um valor inicial, ento o nome da varivel existe no escopo exterior. Isso pode parecer um conflito com Rust no ter null. No entanto, se tentarmos usar uma varivel antes de atribuir um valor a ela, ns teremos um erro em tempo de compilao. Tente!

Quando compilarmos esse cdigo, ns teremos um erro:
error: `x` does not live long enough
   |
6  |         r = &x;
   |              - borrow occurs here
7  |     }
   |     ^ `x` dropped here while still borrowed
...
10 | }
   | - borrowed value needs to live until here

error: `x` does not live long enough
   |
6  |         r = &x;
   |              - borrow occurs here
7  |     }
   |     ^ `x` dropped here while still borrowed
...
10 | }
   | - borrowed value needs to live until here
A varivel x no vive o suficiente. Por que no? Bem, x vai sair de escopo quando passarmos pela chaves na linha 7, terminando o escopo interior. Mas r vlida para o escopo exterior; seu escopo maior e dizemos que ela vive mais tempo. Se Rust permitisse que esse cdigo funcionasse, r estaria fazendo uma referncia memria que foi desalocada quando x saiu de escopo, e qualquer coisa que tentssemos fazer com r no funcionaria corretamente. Ento como o Rust determina que esse cdigo no deve ser permitido?
O Verificador de Emprstimos
A parte do compilador chamada de verificador de emprstimos compara escopos para determinar que todos os emprstimos so vlidos. A Listagem 10-19 mostra o mesmo exemplo da Listagem 10-18 com anotaes mostrando os tempos de vida das variveis.
{
    let r;         // -------+-- 'a
                   //        |
    {              //        |
        let x = 5; // -+-----+-- 'b
        r = &x;    //  |     |
    }              // -+     |
                   //        |
    println!("r: {}", r); // |
                   //        |
                   // -------+
}

{
    let r;         // -------+-- 'a
                   //        |
    {              //        |
        let x = 5; // -+-----+-- 'b
        r = &x;    //  |     |
    }              // -+     |
                   //        |
    println!("r: {}", r); // |
                   //        |
                   // -------+
}
Listagem 10-19: Anotaes de tempos de vida de r e x, chamadas de a e b respectivamente
Ns anotamos o tempo de vida de r com a e o tempo de vida de x com b. Como voc pode ver, o bloco interior de 'b bem menor que o bloco de tempo de vida do exterior 'a'. Em tempo de compilao, o Rust compara o tamanho dos dois tempos de vida e v que r tem um tempo de vida de 'a, mas que ele se refere a um objeto com um tempo de vida 'b. O programa rejeitado porque o tempo de vida de 'b mais curto que o tempo de vida de 'a: o sujeito da referncia no vive tanto quanto a referncia.
Vamos olhar para o exemplo na Listagem 10-20 que no tenta fazer uma referncia solta e compila sem nenhum erro:
{
    let x = 5;            // -----+-- 'b
                          //      |
    let r = &x;           // --+--+-- 'a
                          //   |  |
    println!("r: {}", r); //   |  |
                          // --+  |
}                         // -----+

{
    let x = 5;            // -----+-- 'b
                          //      |
    let r = &x;           // --+--+-- 'a
                          //   |  |
    println!("r: {}", r); //   |  |
                          // --+  |
}                         // -----+
Listagem 10-20: Uma referncia vlida porque os dados tm um tempo de vida maior do que o da referncia
Aqui, x tem o tempo de vida de 'b, que nesse caso tem um tempo de vida maior que o de 'a. Isso quer dizer que r pode referenciar x: o Rust sabe que a referncia em r ser sempre vlida enquanto x for vlido.
Agora que mostramos onde os tempos de vida de referncias esto em um exemplo concreto e discutimos como Rust analisa tempos de vida para garantir que referncias sempre sero vlidas, vamos falar sobre tempos de vidas genricos de parmetros e retornar valores no contexto das funes.
Tempos de Vida Gnericos em Funes

Vamos escrever uma funo que retornar a mais longa de dois cortes de string. Ns queremos ser capazes de chamar essa funo passando para ela dois cortes de strings, e queremos que retorne uma string. O cdigo na Listagem 10-21 deve imprimir A string mais longa abcd uma vez que tivermos implementado a funo maior:

fn main() {
    let string1 = String::from("abcd");
    let string2 = "xyz";

    let resultado = maior(string1.as_str(), string2);

Ns queremos que uma funo seja capaz de aceitar cortes de uma String (que o tipo de varivel string1) assim como literais de string (que o que a varivel strin2 contm).

Enquanto estamos definindo essa funo, no sabemos os valores concretos que sero passados para essa funo, ento no sabemos se o caso if ou o caso else ser executado. Ns tambm no sabemos os tempos de vida concretos das referncias que sero passadas, ento no podemos olhar para esses escopos como fizemos nas Listagem 10-19 e 10-20 afim de determinar que a referncia que retornaremos sempre ser vlida. O verificador de emprstimos no consegue determinar isso tambm porque no sabe como os tempos de vida de x e y se relacionam com o tempo de vida do valor de retorno. Ns vamos adicionar parmetros genricos de tempo de vida que definiro a relao entre as referncias para que o verificador de emprstimos possa fazer sua anlise.
Sintaxe de Anotao de Tempo de Vida

Anotaes de tempo de vida no mudam quanto tempo qualquer uma das referncias envolvidas vivero. Do mesmo modo que funes podem aceitar qualquer tipo de assinatura que especifica um parmetro de tipo genrico, funes podem aceitar referncias com qualquer tempo de vida quando a assinatura especificar um parmetro genrico de tempo de vida. O que anotaes de tempo de vida fazem relacionar os tempos de vida de mltiplas referncias uns com os outros.

Anotaes de tempo de vida tem uma sintaxe levemente incomum: os nomes dos parmetros de tempos de vida precisam comear com uma apstrofe '. Os nomes dos parmetros dos tempos de vida so usualmente todos em caixa baixa, e como tipos genricos, seu nome usualmente so bem curtos. 'a o nome que a maior parte das pessoas usam por padro. Parmetros de anotaes de tempos de vida vo depois do & de uma referncia, e um espao separa a anotao de tempo de vida do tipo da referncia.

Aqui vo alguns exemplos: ns temos uma referncia para um i32 sem um parmetro tempo de vida, uma referncia para um i32 que tem um parmetro de tempo de vida chamado 'a:

&i32        // uma referncia
&'a i32     // uma referncia com um tempo de vida explcito
&'a mut i32 // uma referncia mutvel com um tempo de vida explcito

&i32        // uma referncia
&'a i32     // uma referncia com um tempo de vida explcito
&'a mut i32 // uma referncia mutvel com um tempo de vida explcito
Uma anotao de tempo de vida por si s no tem muito significado: anotaes de tempos de vida dizem ao Rust como os parmetros genricos de tempos de vida de mltiplas referncias se relacionam uns com os outros. Se tivermos uma funo com o parmetro primeiro que uma referncia para um i32 que tem um tempo de vida de 'a, e a funo tem outro parmetro chamado segundo que outra referncia para um i32 que tambm possui um tempo de vida 'a, essas duas anotaes de tempo de vida com o mesmo nome indicam que as referncias primeiro e segundo precisam ambas viver tanto quanto o mesmo tempo de vida genrico.
Anotaes de Tempo de Vida em Assinaturas de Funes

Vamos olhar para anotaes de tempo de vida no contexto da funo maior que estamos trabalhando. Assim como parmetros de tipos genricos, parmetros de tempos de vida genricos precisam ser declarados dentro de colchetes angulares entre o nome da funo e a lista de parmetros. A limitano que queremos dar ao Rust que para as referncias nos parmetros e o valor de retorno devem ter o mesmo tempo de vida, o qual nomearemos 'a e adicionaremos para cada uma das referncias como mostrado na Listagem 10-23:

Nome do Arquivo: src/main.rs

fn maior<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}

fn maior<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() {
        x
    } else {
        y
    }
}
Listagem 10-23: A definio da funo maior especifica todas as referncias na assinatura como tendo o mesmo tempo de vida, 'a
Isso compilar e produzir o resultado que queremos quando usada com a funo main na Listagem 10-21.
A assinatura de funo agora diz que pra algum tempo de vida 'a, a funo receber dois parmetros, ambos sero cortes de string que vivem pelo menos tanto quanto o tempo de vida 'a. A funo retornar um corte de string que tambm vai durar tanto quanto o tempo de vida 'a. Esse o contrato que estamos dizendo ao Rust que queremos garantir.
Especificando os parmetros de tempo de vida nessa assinatura de funo, no estamos modificando os tempos de vida de quaisquer valores passados ou retornados, mas estamos dizendo que quaisqueres valores que no concordem com esse contrato devem ser rejeitados pelo verificador de emprstimos. Essa funo no sabe (ou no precisa saber) exatamente quanto tempo x e y vo viver, apenas precisa saber que existe algum escopo que pode ser substitudo por 'a que ir satisfazer essa assinatura.
Quando estiver anotando tempos de vidas em funes, as anotaes vo na assinatura da funo, e no no cdigo no corpo da funo. Isso acontece porque o Rust consegue analisar o cdigo dentro da funo sem nenhuma ajuda, mas quando uma funo tem referncias para ou de cdigos de fora daquela funo, os tempos de vida dos argumentos ou os valores de retorno podero ser diferentes cada vez que a funo chamada. Isso seria incrivelmente custoso e frequentemente impossvel para o Rust descobrir. Nesse caso, precisamos anotar os tempos de vida ns mesmos.
Quando referncias concretas so passadas para maior, o tempo de vida concreto que substitudo por 'a a parte do escopo de x que sobrepe o escopo de y. J que escopos sempre se aninham, outra maneira de dizer isso que o tempo de vida genrico 'a ter um tempo de vida concreto igual ao menor dos tempos de vida de x e y. Porque ns anotamos a referncia retornada com o mesmo parmetro 'a, a referncia retornada ser portanto garantida de ser vlida tanto quanto for o tempo de vida mais curto de x e y.
Vamos ver como isso restringe o uso da funo maior passando referncias que tem diferentes tempos de vida concretos. A Listagem 10-25 um exemplo direto que deve corresponder suas intuies de qualquer linguagem: string1 vlida at o final do escopo exterior, strin2 vlida at o final do escopo, a string2 vlida at o final do escopo interior. Com o verificador de emprstimos aprovando esse cdigo; ele vai compilar e imprimir A string mais longa :

Pensando em Termos de Tempos de Vida

O modo exato de especificar parmetros de tempos de vida depende do que sua funo est fazendo. Por exemplo, se mudaramos a implementao da funo maior para sempre retornar o primeiro argumento ao invs do corte de string mais longo, no precisaramos especificar um tempo de vida no parmetro y. Este cdigo compila:

Nome do arquivo: src/main.rs

fn longest<'a>(x: &'a str, y: &str) -> &'a str {
    x
}

fn longest<'a>(x: &'a str, y: &str) -> &'a str {
    x

Os padres programados nas anlises de referncia de Rust so chamados de regras de eliso de tempo de vida. Essas no so regras para o programador seguir; as regras so um conjunto de casos particular que o compilador ir considerar, e se seu cdigo se encaixa nesses casos, voc no precisa escrever os tempos de vida explicitamente.
As regras de eliso no fornecem total inferncia: se o Rust aplicar as regras de forma determinstica ainda podem haver ambiguidades como quais tempos de vida as referncias restantes deveriam ter. Nesse caso, o compilador dar um erro que pode ser solucionado adicionando anotaes de tempo de vida que correspondem com as suas intenes para como as referncias se relacionam umas com as outras.
Primeiro, algumas definies: Tempos de vida em parmetros de funes ou mtodos so chamadas tempos de vida de entrada, e tempos de vida em valores de retorno so chamados de tempos de vida de sada.
Agora, as regras que o compilador usa para descobrir quais referncias de tempos de vidas tm quando no h anotaes explcitas. A primeira regra se aplica a tempos de vida de entrada, e a segunda regra se aplica a tempos de vida de sada. Se o compilador chega no fim das trs regras e ainda h referncias que ele no consegue descobrir tempos de vida, o compilador ir parar com um erro.

  1. Cada parmetro que uma referncia tem seu prprio parmetro de tempo de vida. O Tempo de Vida Esttico
H um tipo especial de tempo de vida que precisamos discutir: 'static. O tempo de vida static a durao completa do programa. Todos os literais de string tm um tempo de vida static, o qual podemos escolher anotar como o seguinte:

Voc pode ver sugestes de usar o tempo de vida 'static em uma mensagem de ajuda de erro, mas antes de especificar 'static como o tempo de vida para uma referncia, pense sobre se a referncia que voc tem uma que vive todo o tempo de vida do seu programa ou no (ou mesmo se voc quer que ele viva tanto, se poderia). Na maior parte do tempo, o problma no cdigo uma tentativa de criar uma referncia solta ou uma incompatibilidade dos tempos de vida disponveis, e a soluo consertar esses problemas, no especificar um tempo de vida 'static.
