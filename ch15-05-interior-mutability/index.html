RefCell<T> e a Pattern de Mutabilidade Interior

Mutabilidade interior (interior mutability) uma design pattern em Rust que lhe permite modificar um dado mesmo quando h referncias imutveis a ele: normalmente, esta ao proibida pelas regras de emprstimo. Para fazer isso, a pattern usa cdigo unsafe (inseguro) dentro de uma estrutura de dados para dobrar as regras normais do Rust que governam mutao e emprstimo. Ns ainda no cobrimos cdigo unsafe; faremos isso no Captulo 19. Podemos usar tipos que usam a pattern de mutabilidade interior quando podemos garantir que as regras de emprstimo sero seguidas em tempo de execuo, ainda que o compilador no o possa garantir. O cdigo unsafe envolvido ento embrulhado em uma API safe, e o tipo exterior permanece imutvel.

Para explorar este conceito, vamos ver o tipo RefCell<T> que segue a pattern de mutabilidade interior.

Aplicando Regras de Emprstimo em Tempo de Execuo com o RefCell<T>

Diferente do Rc<T>, o tipo RefCell<T> representa posse nica sobre o dado que ele contm. Ento o que torna o RefCell<T> diferente de um tipo como o Box<T>? Lembre-se das regras de emprstimo que voc aprendeu no Captulo 4:

  • Em qualquer momento, voc pode ter um dos mas no ambos os seguintes: uma nica referncia mutvel ou qualquer nmero de referncias imutveis;
  • Referncias devem sempre ser vlidas.

Com referncias e com o Box<T>, as invariantes das regras de emprstimo so aplicadas em tempo de compilao. Com o RefCell<T>, essas invariantes so aplicadas em tempo de execuo. Com referncias, se voc quebra essas regras, voc recebe um erro de compilao. Com o RefCell<T>, se voc quebrar essas regras, seu programa ir sofrer um panic! e terminar.

As vantagens de checar as regras de emprstimo em tempo de compilao so que erros so pegos mais cedo no processo de desenvolvimento, e no h nenhum custo de desempenho de execuo porque toda a anlise completada de antemo. Por esses motivos, checar as regras de emprstimo em tempo de compilao a melhor opo na maioria dos casos, e por isso este o padro do Rust.

A vantagem de checar as regras de emprstimo em tempo de execuo, alternativamente, que certos cenrios memory-safe (seguros em termos de memria) so ento permitidos, ao passo que seriam proibidos pelas checagens em tempo de compilao. A anlise esttica, como a do compilador Rust, inerentemente conservadora. Algumas propriedades do programa so impossveis de detectar analisando o cdigo: o exemplo mais famoso o Problema da Parada, que est alm do escopo deste livro mas um tpico interessante para pesquisa.

Como algumas anlises so impossveis, se o compilador Rust no consegue se assegurar que o cdigo obedece s regras de posse, ele pode rejeitar um programa correto; neste sentido, ele conservador. Se o Rust aceitasse um programa incorreto, os usurios no poderiam confiar nas garantias que ele faz. Se, por outro lado, o Rust rejeita um programa correto, o programador ter alguma inconvenincia, mas nada catastrfico pode acontecer. O tipo RefCell<T> til quando voc tem certeza que seu cdigo segue as regras de emprstimo, mas o compilador incapaz de entender e garantir isso.

Assim como o Rc<T>, o RefCell<T> apenas para uso em cenrios de thread nica e lhe daro um erro de compilao se voc tentar us-lo em um contexto de mltiplas threads. Falaremos sobre como obter a funcionalidade de um RefCell<T> em um programa multithread no Captulo 16.

Aqui est uma recapitulao das razes para escolher o Box<T>, o Rc<T> ou o RefCell<T>:

  • O Rc<T> permite mltiplos possuidores do mesmo dado; Box<T> e RefCell<T> tm possuidores nicos.
  • O Box<T> permite emprstimos imutveis ou mutveis checados em tempo de compilao; o Rc<T> permite apenas emprstimos imutveis em tempo de compilao; o RefCell<T> permite emprstimos imutveis ou mutveis checados em tempo de execuo.
  • Como o RefCell<T> permite emprstimos mutveis checados em tempo de execuo, ns podemos modificar o valor dentro de um RefCell<T> mesmo quando o RefCell<T> imutvel.

Modificar o valor dentro de um valor imutvel a pattern de mutabilidade interior. Vamos dar uma olhada em uma situao em que a mutabilidade interior til e examinar como ela possvel.

Um Caso de Uso para a Mutabilidade Interior: Objetos Simulados

Um dubl de teste (test double) um conceito geral de programao para um tipo usado no lugar de outro durante os testes. Objetos simulados (mock objects) so tipos especficos de dubls de teste que registram o que acontece durante o teste para que possamos confirmar que as aes corretas aconteceram.

Rust no tem objetos da mesma forma que outras linguagens, e no tem funcionalidade de objetos simulados embutida na biblioteca padro como algumas outras linguagens. Contudo, certamente podemos criar uma struct que serve os mesmos propsitos que um objeto simulado.

Eis o cenrio que vamos testar: vamos criar uma biblioteca que acompanha um valor contra um valor mximo e envia mensagens com base em quo prximo do valor mximo o valor atual est. Esta biblioteca pode ser usada para acompanhar a cota de um usurio para o nmero de chamadas de API que ele tem direito a fazer, por exemplo.

Nossa biblioteca ir prover somente a funcionalidade de acompanhar quo perto do mximo um valor est e o que as mensagens deveriam ser em quais momentos. As aplicaes que usarem nossa biblioteca tero a responsabilidade de prover o mecanismo para enviar as mensagens: a aplicao pode pr a mensagem na prpria aplicao, enviar um email, uma mensagem de texto, ou alguma outra coisa. A biblioteca no precisa saber deste detalhe. Tudo que ela precisa de algo que implemente uma trait que iremos prover chamada Mensageiro. A Listagem 15-20 mostra o cdigo da biblioteca:

Arquivo: src/lib.rs

pub trait Mensageiro {
    fn enviar(&self, msg: &str);
}

pub struct AvisaLimite<'a, T: 'a + Mensageiro> {
    mensageiro: &'a T,
    valor: usize,
    max: usize,
}

impl<'a, T> AvisaLimite<'a, T>
    where T: Mensageiro {
    pub fn new(mensageiro: &T, max: usize) -> AvisaLimite<T> {
        AvisaLimite {
            mensageiro,
            valor: 0,
            max,
        }
    }

    pub fn set_valor(&mut self, valor: usize) {
        self.valor = valor;

        let porcentagem_do_max = self.valor as f64 / self.max as f64;

        if porcentagem_do_max >= 0.75 && porcentagem_do_max < 0.9 {
            self.mensageiro.enviar("Aviso: Voc usou mais de 75% da sua cota!");
        } else if porcentagem_do_max >= 0.9 && porcentagem_do_max < 1.0 {
            self.mensageiro.enviar("Aviso urgente: Voc usou mais de 90% da sua cota!");
        } else if porcentagem_do_max >= 1.0 {
            self.mensageiro.enviar("Erro: Voc excedeu sua cota!");
        }
    }
}

O RefCell<T> Registra Emprstimos em Tempo de Execuo

Quando estamos criando referncias imutveis e mutveis, usamos as sintaxes & e &mut, respectivamente. Com o RefCell<T>, usamos os mtodos borrow e borrow_mut, que so parte da API safe que pertence ao RefCell<T>. O mtodo borrow retorna o ponteiro inteligente Ref<T>, e o borrow_mut retorna o ponteiro inteligente RefMut<T>. Ambos os tipos implementam Deref, ento podemos trat-los como referncias normais.

O tipo RefCell<T> mantm registro de quantos ponteiros inteligentes Ref<T> e RefMut<T> esto atualmente ativos. Cada vez que chamamos borrow, o RefCell<T> aumenta seu contador de quantos emprstimos imutveis esto ativos. Quando um valor Ref<T> sai de escopo, o contador de emprstimos imutveis diminui em um. Assim como as regras de emprstimo em tempo de compilao, o RefCell<T> nos permite ter vrios emprstimos imutveis ou um emprstimo mutvel em um dado momento.

Se tentarmos violar estas regras, em vez de receber um erro do compilador como iramos com referncias, a implementao de RefCell<T> chamar panic! em tempo de execuo. A Listagem 15-23 mostra uma modificao da implementao do enviar da Listagem 15-22. Estamos deliberadamente tentando criar dois emprstimos mutveis ativos para o mesmo escopo para ilustrar que o RefCell<T> nos impede de fazer isto em tempo de execuo:

Arquivo: src/lib.rs

impl Mensageiro for MensageiroSimulado {
    fn enviar(&self, mensagem: &str) {
        let mut emprestimo_um = self.mensagens_enviadas.borrow_mut();
        let mut emprestimo_dois = self.mensagens_enviadas.borrow_mut();

        emprestimo_um.push(String::from(mensagem));
        emprestimo_dois.push(String::from(mensagem));
    }
}

    }

Listagem 15-23: Criando duas referncias mutveis no mesmo escopo para ver que o RefCell<T> ir entrar em pnico (i.e., executar panic!)
Ns criamos uma varivel emprestimo_um para o ponteiro inteligente RefMut<T> retornado por borrow_mut. Ento criamos outro emprstimo mutvel da mesma forma na varivel emprestimo_dois. Isto resulta em duas referncias mutveis no mesmo escopo, o que no permitido. Pegar erros de emprstimo em tempo de execuo em vez de em tempo de compilao significa encontrar defeitos no nosso cdigo mais tarde no processo de desenvolvimento, e possivelmente nem mesmo at que nosso cdigo j tenha sido implantado em produo. Alm disso, nosso cdigo ir incorrer em uma pequena penalidade de desempenho de execuo como resultado de manter registro dos emprstimos em tempo de execuo em vez de compilao. Ainda assim, usar o RefCell<T> nos torna possvel escrever um objeto simulado que pode se modificar para registrar as mensagens que ele j viu enquanto o usamos em um contexto onde apenas valores imutveis so permitidos. Podemos usar o RefCell<T>, apesar de seus trade-offs, para obter mais funcionalidade do que referncias regulares nos do.
Conseguindo Mltiplos Possuidores de Dados Mutveis pela Combinao de Rc<T> e RefCell<T>

Um jeito comum de usar o RefCell<T> em combinao com o Rc<T>. Lembre-se de que o Rc<T> nos permite ter mltiplos possuidores de algum dado, mas ele s nos permite acesso imutvel a esse dado. Se temos um Rc<T> que contm um RefCell<T>, podemos ter um valor que pode ter mltiplos possuidores e que podemos modificar!

Por exemplo, lembre-se da cons list na Listagem 15-18 onde usamos o Rc<T> para nos permitir que mltiplas listas compartilhassem posse de outra lista. Como o Rc<T> guarda apenas valores imutveis, ns no podemos modificar nenhum dos valores na lista uma vez que os criamos. Vamos adicionar o RefCell<T> para ganhar a habilidade de mudar os valores nas listas. A Listagem 15-24 mostra que, usando um RefCell<T> na definio do Cons, podemos modificar o valor armazenado em todas as listas:

Arquivo: src/main.rs

#[derive(Debug)]
enum List {
    Cons(Rc<RefCell<i32>>, Rc<List>),
    Nil,
}

use List::{Cons, Nil};
use std::rc::Rc;
use std::cell::RefCell;

fn main() {
    let valor = Rc::new(RefCell::new(5));

    let a = Rc::new(Cons(Rc::clone(&valor), Rc::new(Nil)));

    let b = Cons(Rc::new(RefCell::new(6)), Rc::clone(&a));
    let c = Cons(Rc::new(RefCell::new(10)), Rc::clone(&a));

    *valor.borrow_mut() += 10;

    println!("a depois = {:?}", a);
    println!("b depois = {:?}", b);
    println!("c depois = {:?}", c);
}

#[derive(Debug)]
enum List {
    Cons(Rc<RefCell<i32>>, Rc<List>),
    Nil,
}

use List::{Cons, Nil};
use std::rc::Rc;
use std::cell::RefCell;

fn main() {
    let valor = Rc::new(RefCell::new(5));

    let a = Rc::new(Cons(Rc::clone(&valor), Rc::new(Nil)));

    let c = Cons(Rc::new(RefCell::new(10)), Rc::clone(&a));

    println!("a depois = {:?}", a);
    println!("b depois = {:?}", b);
    println!("c depois = {:?}", c);
}
Listagem 15-24: Usando Rc<RefCell<i32>> para criar uma List que podemos modificar
Ns criamos um valor que uma instncia de Rc<RefCell<i32>> e o armazenamos em uma varivel chamada valor para que possamos acess-lo diretamente mais tarde. Ento criamos uma List em a com uma variante Cons que guarda valor.

Ns embrulhamos a lista a em um Rc<T> para que, quando criarmos as listas b e c, elas possam ambas se referir a a, que o que fizemos na Listagem 15-18.

Depois de criarmos as listas em a, b e c, adicionamos 10 ao valor em valor. Fazemos isto chamando borrow_mut em valor, o que usa a funcionalidade de desreferncia automtica que discutimos no Captulo 5 (veja a seo Onde est o operador ->?) para desreferenciar o Rc<T> ao valor interno RefCell<T>. O mtodo borrow_mut retorna um ponteiro inteligente RefMut<T> no qual usamos o operador de desreferncia e modificamos o valor interno.

Quando imprimimos a, b e c, podemos ver que todos eles tm o valor modificado de 15 em vez de 5:

a depois = Cons(RefCell { value: 15 }, Nil)
b depois = Cons(RefCell { value: 6 }, Cons(RefCell { value: 15 }, Nil))
c depois = Cons(RefCell { value: 10 }, Cons(RefCell { value: 15 }, Nil))

a depois = Cons(RefCell { value: 15 }, Nil)
b depois = Cons(RefCell { value: 6 }, Cons(RefCell { value: 15 }, Nil))
c depois = Cons(RefCell { value: 10 }, Cons(RefCell { value: 15 }, Nil))
Esta tcnica bem bacana! Usando um RefCell<T>, temos uma List exteriormente imutvel. Mas podemos usar os mtodos no RefCell<T> que do acesso a sua mutabilidade interior para que possamos modificar nossos dados quando precisarmos. As checagens em tempo de execuo das regras de emprstimo nos protegem de corridas de dados, e s vezes vale a pena trocar um pouco de velocidade por esta flexibilidade nas nossas estruturas de dados.
A biblioteca padro tem outros tipos que proveem mutabilidade interior, como o Cell<T>, que parecido, exceto que em vez de dar referncias ao valor interno, o valor copiado para dentro e para fora do Cell<T>. Tem tambm o Mutex<T>, que oferece mutabilidade interior que segura de usar entre threads; vamos discutir seu uso no Captulo 16. Confira a documentao da biblioteca padro para mais detalhes sobre as diferenas entre estes tipos.
